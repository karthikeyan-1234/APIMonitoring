02 — Conventions
Naming Conventions
Every name flows from a single source of truth. The Docker image tag you pass to docker build -t must appear verbatim in the YAML image: field. The K8s Service metadata.name becomes the DNS name Kong resolves. Getting this chain wrong is the single most common cause of pod crashes and 502 errors.

⚠ The #1 Mistake — image tag mismatch

Building with docker build -t api1-app:stable . but writing image: api1-app:v1 in the YAML causes the pod to fail. Kubernetes looks for the exact tag in the local Docker image store. With imagePullPolicy: IfNotPresent, if the tag doesn't exist locally, the pod enters ImagePullBackOff. Always build and reference the same tag.

03 — Reference
Kong Concepts Deep Dive
Three core Kong objects handle every request: Service, Route, and Upstream. Understanding exactly what each field does — and where its value comes from — prevents the most common configuration mistakes.

Kong Service

A Kong Service is the outbound connection definition — it tells Kong where to forward a matched request. Its host field is either a direct K8s DNS name (Blue/Green strategy) or a Kong Upstream name (Canary strategy). These are fundamentally different; one is DNS-resolved, the other is resolved via Kong's upstream registry.

host

Kong Route — Full Field Anatomy

A Route is the incoming request matcher. Kong evaluates all routes and forwards the request to the Service of the first matching route. Understanding every field prevents silent mismatch bugs.

FieldValue in this setupMeaning
nameapi1-test-routeUnique identifier. Used in Admin API calls. Convention: {app}-{path}-route.
paths["/test"]URL prefix Kong listens for on the Proxy. Requests whose path starts with this value are matched.
strip_pathfalse for /test · true for /bgtestfalse = full path forwarded as-is. true = route prefix stripped before forwarding; Service.path then prepends the app path.
headersnull (normal) · {"X-Canary":["true"]} (header routing)If set, only requests carrying this exact header and value match this route. Used for header-based traffic targeting.
regex_priority0 (default) · 10 (header route)When multiple routes match the same path, the highest priority wins. Header routes must be higher than the base route.
protocols["http","https"]Kong accepts both protocols. Default when not specified.
hostsnullNull = match any Host header. Set to ["vault.local"] to restrict to a specific virtual host.
methodsnullNull = all HTTP methods accepted. Set to ["GET","POST"] to restrict.
preserve_hostfalsefalse = Kong sets Host header to upstream hostname. true = original client Host header preserved.
path_handlingv0Default. Handles path prefix stripping correctly for standard cases.
request_bufferingtrueSet false for streaming uploads or large request bodies.
response_bufferingtrueSet false for SSE or streaming responses.
ℹ strip_path + Service.path in action

Route: /bgtest · strip_path: true · Service.path: /test
Client sends GET /bgtest → Kong strips /bgtest leaving / → prepends Service.path → upstream receives GET /test

Route: /test · strip_path: false · no Service.path
Client sends GET /test → Kong preserves path → upstream receives GET /test

Kong Upstream & Targets

An Upstream is Kong's built-in load balancer. It acts as a virtual hostname referenced by a Kong Service. Targets are the actual backend K8s service DNS addresses with weights. Adjusting target weights is how canary traffic shifting works — no service or route changes needed.

ObjectFieldValueMeaning
Upstreamnameapi1-upstreamUsed verbatim as the host in the Kong Service. Must match exactly — Kong resolves this name through its upstream registry, not DNS.
algorithmround-robinRequest distribution algorithm. Options: round-robin, least-connections, consistent-hashing.
Targettargetapi1-stable-service.default.svc.cluster.local:80Full K8s FQDN + port. Must be reachable from inside the cluster. Kong resolves this via cluster DNS.
weight70 / 30Relative weight. 70+30=100 → 70% of requests go to stable, 30% to canary. Set to 0 to drain a target without deleting it.
✓ The key architectural difference between strategies

Canary: Kong Service host = api1-upstream → routes through Kong's load balancer across multiple K8s services by weight.
Blue/Green: Kong Service host = api1-blue-service.default.svc.cluster.local → routes directly to one K8s service. Cutover = PATCH host to the green K8s service DNS name.

Step 4 — Configure Kong

The complete Kong object chain for Canary is: Kong Service → Kong Route → Upstream → Targets. Starting from a clean slate, we create all four objects in order: first the Service and its Route (so traffic can flow), then the Upstream and Targets (to enable load balancing), and finally we redirect the Service to point at the Upstream.

host = stable K8s DNS initially
→
② Create Kong Route

③ Create Upstream
+ add Targets

④ PATCH Kong Service

⑤ Verify Route

1
Create the Kong Service — initially pointing at the Stable K8s service

This Kong Service is the outbound connection definition. It starts pointing directly at the Stable K8s service DNS. Later (Step 4) we PATCH its host to point at the Upstream instead, activating load balancing. Creating it first allows us to immediately attach a Route in Step 2.

▶PowerShell POST /services — create api1-service targeting stable K8s service directly 
Invoke-RestMethod -Uri "http://kong-admin.local/services" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        name     = "api1-service"
        host     = "api1-stable-service.default.svc.cluster.local"
        port     = 80
        protocol = "http"
    } | ConvertTo-Json)

# Confirm the service was created correctly
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-service" |
    Select-Object name, host, port, protocol | Format-List
2
Create the Kong Route on /test — attached to api1-service

A Route is the incoming request matcher. This route listens for any request whose path starts with /test and forwards it to api1-service. strip_path: false means the full /test path is forwarded as-is to the upstream. This route is created once and never needs to change — only the Service's host field changes when we activate the Upstream in Step 4.

▶PowerShell POST /services/api1-service/routes — api1-test-route on /test 
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-service/routes" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        name       = "api1-test-route"
        paths      = @("/test")
        strip_path = $false     # forward full /test path to upstream as-is
    } | ConvertTo-Json)

# Confirm the route was created and linked to the correct service
Invoke-RestMethod -Uri "http://kong-admin.local/routes/api1-test-route" | ConvertTo-Json -Depth 4
3
Create the Kong Upstream

The Upstream is Kong's virtual load balancer. Its name is what you'll set as the host in the Kong Service (Step 4). It is not a DNS hostname — Kong resolves it via its internal upstream registry.

▶PowerShell POST /upstreams — create api1-upstream with round-robin 
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        name      = "api1-upstream"
        algorithm = "round-robin"
    } | ConvertTo-Json)

# Confirm it was created
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream" | ConvertTo-Json

Redirect the Kong Service to the Upstream

api1-service was created in Step 1 pointing at the Stable K8s DNS. Now we PATCH only its host field to point at the Upstream name — this activates Kong's load balancer so all traffic flowing through api1-test-route is distributed across Stable and Canary by weight. The route itself is completely untouched.

▶PowerShell PATCH /services/api1-service — point host at Upstream name (activates load balancing) 
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-service" `
    -Method PATCH -ContentType 'application/json' `
    -Body (@{ host = "api1-upstream" } | ConvertTo-Json)

# Confirm the service host is now the upstream name
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-service" |
    Select-Object name, host, port | Format-List
8
Confirm the Route is still intact

Verify that api1-test-route still exists and still references api1-service. The service PATCH does not affect routes.

▶PowerShell GET /routes/api1-test-route — confirm path and service link are unchanged 
Invoke-RestMethod -Uri "http://kong-admin.local/routes/api1-test-route" | ConvertTo-Json -Depth 4
# Confirm: paths=["/test"], service.id matches api1-service

Blue/Green Deployment
Run two full environments simultaneously. Blue is live. Green is fully deployed and privately validated before any public traffic touches it. A single PATCH to the Kong Service's host field switches all traffic instantly — zero downtime, immediate rollback.

ℹ Why /bgtest and not /test?

Blue/Green uses its own dedicated Kong Service (api1-bg-service) and Route (api1-bg-route) on path /bgtest. This keeps it completely isolated from the Canary strategy's /test route. The app's actual endpoint is /test — the route uses strip_path: true and the Kong Service sets path: /test so the upstream receives the correct path.

06 — Strategy
Header-Based Traffic Routing
The safest production routing pattern. A new higher-priority route with a header condition intercepts only requests that carry a specific header — all other traffic continues to the original stable route, completely untouched. Zero risk to existing users.
✓ Core Principle — additive, never disruptive

Create a new Kong Service pointing at the canary/green K8s service, then a new Route on the same path with a headers condition and higher regex_priority. Do not modify the existing stable route or service. Kong evaluates routes by priority — the header route wins only for header-carrying requests; everything else falls through to the original route as before.

Why this is production-safe
  • Existing routes untouchedzero config drift risk
  • Zero downtimepurely additive operation
  • Targeted testingonly flagged clients see new version
  • Instant rollbackDELETE the header route
  • No Upstream neededdirect K8s DNS target
Industry adoption
  • Kongheaders + regex_priority
  • IstioVirtualService header match
  • AWS ALBheader-based listener rules
  • TraefikHeaders middleware routing
  • NGINX$http_x_* variable matching

How Route Priority Works

1
Create a dedicated Kong Service pointing directly at the canary K8s service

This is a direct-to-K8s-service connection, not via an Upstream. It is separate from api1-service — it will not share the route with stable traffic.

▶PowerShell POST /services — create api1-canary-direct pointing at canary K8s service 
Invoke-RestMethod -Uri "http://kong-admin.local/services" `

Create a header-conditioned route at higher priority — do NOT modify the existing stable route

This route matches the same path /test as the stable route but wins because its regex_priority (10) is higher than the stable route's (0). It only activates when the X-Canary: true header is present.

▶PowerShell POST /routes — api1-canary-header-route on /test, X-Canary:true, priority 10 
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-canary-direct/routes" `

Traffic Management
Shift canary Upstream weights in phases without any redeployment. Weight changes take effect on the next request — no Kong restart, no pod restart.

Get & Store Target IDs

Run this once after creating the Upstream and Targets. Save $stableId and $canaryId in your PowerShell session — you'll need them for the weight-shifting commands below.

▶PowerShell Retrieve and store stable/canary target IDs from the Upstream 
$targets  = Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets"
$stableId = ($targets.data | Where-Object { $_.target -like "*stable*" }).id
$canaryId = ($targets.data | Where-Object { $_.target -like "*canary*" }).id

Write-Host "Stable ID : $stableId"
Write-Host "Canary ID : $canaryId"

# Verify — should show both targets with their current weights
$targets.data | Select-Object target, weight, id | Format-Table -AutoSize

Canary Promotion Phases

PhaseStable WeightCanary WeightUse When
Initial exposure9010First release — minimal blast radius
Phase 27030No errors observed at 10%
Phase 35050Confidence growing — equal split
Full promotion0100Canary is now production — stable is idle
Emergency rollback1000Errors detected — revert all traffic instantly
▶PowerShell Set-CanaryWeights helper — run any phase with one function call 
function Set-CanaryWeights {
    param(
        [Parameter(Mandatory)][int]$StableWeight,
        [Parameter(Mandatory)][int]$CanaryWeight
    )

    Invoke-RestMethod `
        -Uri "http://kong-admin.local/upstreams/api1-upstream/targets/$stableId" `
        -Method PATCH -ContentType 'application/json' `
        -Body (@{ weight = $StableWeight } | ConvertTo-Json)

    Invoke-RestMethod `
        -Uri "http://kong-admin.local/upstreams/api1-upstream/targets/$canaryId" `
        -Method PATCH -ContentType 'application/json' `
        -Body (@{ weight = $CanaryWeight } | ConvertTo-Json)

    Write-Host "✓ Stable: $StableWeight  |  Canary: $CanaryWeight"

    # Read back to confirm
    $t = Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets"
    $t.data | Select-Object target, weight | Format-Table -AutoSize
}

# --- Usage ---
Set-CanaryWeights -StableWeight 90  -CanaryWeight 10   # Phase 1 — initial
Set-CanaryWeights -StableWeight 70  -CanaryWeight 30   # Phase 2
Set-CanaryWeights -StableWeight 50  -CanaryWeight 50   # Phase 3 — equal split
Set-CanaryWeights -StableWeight 0   -CanaryWeight 100  # Full promotion
Set-CanaryWeights -StableWeight 100 -CanaryWeight 0    # Emergency rollback
