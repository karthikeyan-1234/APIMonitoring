One source of truth
Naming rules: avoid pod failures
Your Docker image tag must match the image: field in YAML. Use imagePullPolicy: IfNotPresent so Kubernetes never tries to pull from Docker Hub.
▶📊 TableNaming chain (stable example)
docker build -tDeployment image:K8s ServiceKong target host
api1-app:stableapi1-app:stableapi1-stable-serviceapi1-stable-service.default.svc.cluster.local:80
💡 Tip: The pod reads APP_VERSION and DEPLOYMENT_SLOT from environment variables, so the same Docker image can behave differently per deployment.
Kong 101
Services, Routes & Upstreams
A Route matches the incoming URL. It points to a Service that defines the backend (K8s service or upstream). Upstreams let you split traffic between versions.
▶📊 TableRoute fields explained (simple)
FieldMeaning
pathsURL prefix that triggers this route (e.g. /test)
strip_pathIf true, Kong removes the matched prefix before forwarding
headersOnly match if header present (header-based routing)
regex_priorityHigher priority wins when multiple routes match
Strategy 1
Canary: gradual traffic shift
Send 10% of users to the new version, then 30%, then 100%. No redeploy — just update weights in Kong.

📦 1. Build images

▶PowerShellBuild stable & canary
docker build -t api1-app:stable .; docker build -t api1-app:canary .
▶YAMLapi1-stable.yaml (full)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api1-stable
  labels:
    app: api1-app
    version: stable
spec:
  replicas: 1
  selector:
    matchLabels:
      app: api1-app
      version: stable
  template:
    metadata:
      labels:
        app: api1-app
        version: stable
    spec:
      nodeName: desktop-control-plane
      containers:
        - name: api1-app
          image: api1-app:stable
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 80
              name: http
          env:
            - name: ASPNETCORE_ENVIRONMENT
              value: "Development"
            - name: ASPNETCORE_URLS
              value: "http://+:80"
            - name: OTEL_EXPORTER_OTLP_ENDPOINT
              value: "http://otel-collector:4317"
            - name: APP_VERSION
              value: "v1-stable"
            - name: DEPLOYMENT_SLOT
              value: "stable"
            - name: POD_IP
              valueFrom:
                fieldRef:
                  fieldPath: status.podIP
          livenessProbe:
            httpGet: { path: /test, port: 80 }
            initialDelaySeconds: 10
            periodSeconds: 10
          readinessProbe:
            httpGet: { path: /test, port: 80 }
            initialDelaySeconds: 5
            periodSeconds: 5
---
apiVersion: v1
kind: Service
metadata:
  name: api1-stable-service
spec:
  type: ClusterIP
  selector:
    app: api1-app
    version: stable
  ports:
    - port: 80
      targetPort: 80
      name: http

⚙️ 2. Kong upstream + targets

▶PowerShellCreate upstream & targets (70% stable / 30% canary)
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams" -Method POST -Body (@{name="api1-upstream"} | ConvertTo-Json) -ContentType 'application/json'
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets" -Method POST -Body (@{target="api1-stable-service.default.svc.cluster.local:80"; weight=70} | ConvertTo-Json)
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets" -Method POST -Body (@{target="api1-canary-service.default.svc.cluster.local:80"; weight=30} | ConvertTo-Json)
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-service" -Method PATCH -Body (@{host="api1-upstream"} | ConvertTo-Json)

Strategy 3
Header‑based routing (safest)
Add a new route with a header condition and higher priority. Existing stable routes stay untouched. Only requests with the special header go to the new version.

🎯 Canary via header

▶PowerShellHeader route for canary (X-Canary: true)
Invoke-RestMethod -Uri "http://kong-admin.local/services" -Method POST -Body (@{name="api1-canary-direct"; host="api1-canary-service.default.svc.cluster.local"; port=80} | ConvertTo-Json)
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-canary-direct/routes" -Method POST -Body (@{name="api1-canary-header-route"; paths=@("/test"); headers=@{"X-Canary"=@("true")}; regex_priority=10} | ConvertTo-Json)
▶PowerShellTest: with and without header
Invoke-RestMethod "http://kong-proxy.local/test"                              # stable
Invoke-RestMethod "http://kong-proxy.local/test" -Headers @{"X-Canary"="true"} # canary

🟢 Blue/Green via header

▶PowerShellX-Slot: green route (priority 10)
Invoke-RestMethod -Uri "http://kong-admin.local/services" -Method POST -Body (@{name="api1-green-direct"; host="api1-green-service.default.svc.cluster.local"; port=80; path="/test"} | ConvertTo-Json)
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-green-direct/routes" -Method POST -Body (@{name="api1-green-header-route"; paths=@("/bgtest"); headers=@{"X-Slot"=@("green")}; regex_priority=10; strip_path=$true} | ConvertTo-Json)

Manage traffic
Canary weight phases
Update weights without restarting anything.
▶PowerShellPromotion helper (get target IDs first)
$targets = Invoke-RestMethod "http://kong-admin.local/upstreams/api1-upstream/targets"
$stableId = ($targets.data | Where-Object { $_.target -like "*stable*" }).id
$canaryId = ($targets.data | Where-Object { $_.target -like "*canary*" }).id
function Set-CanaryWeights($stableWeight, $canaryWeight) {
  Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets/$stableId" -Method PATCH -Body (@{weight=$stableWeight}|ConvertTo-Json)
  Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets/$canaryId" -Method PATCH -Body (@{weight=$canaryWeight}|ConvertTo-Json)
}
