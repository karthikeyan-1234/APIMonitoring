Every name in this system flows from a single source. Understanding this chain prevents the most common mistakes — especially the image tag mismatch that causes pods to fail on startup.

Kong target

⚠ The #1 Mistake

Building with docker build -t api1-app:stable . but writing image: api1-app:v1 in the YAML. Kubernetes looks for the exact tag. If it doesn't match a local image and imagePullPolicy: Always is set, it tries Docker Hub — and fails. Always use imagePullPolicy: IfNotPresent for local images.

03 — Reference
Kong Concepts Deep Dive
Kong's three core objects — Service, Route, and Upstream — work together. Every field has a specific source. This section documents exactly where each value comes from in your local setup.

Kong Service

A Kong Service represents your upstream backend. It is the bridge between a Kong Route (the incoming request) and the actual server that handles it. In our setup, a Service either points to a direct K8s service DNS name (Blue/Green) or to a Kong Upstream name (Canary).

Service → Direct K8s (Blue/Green)
name
api1-bg-serviceYou choose this. Referenced when creating routes.
host
api1-blue-service.default.svc.cluster.localK8s Service name + namespace + cluster suffix. This is the PATCH target during cutover.
port
80From K8s Service spec.ports[].port
protocol
httpKestrel is on plain HTTP internally
path
/testSet when your Kong route path ≠ app path. Here /bgtest → /test.
Service → Upstream (Canary)
name
api1-canary-svcYou choose this. Referenced when creating routes.

api1-upstreamThe Kong Upstream name — NOT a DNS name. Kong resolves via its upstream registry.
port

protocol
httpKestrel is on plain HTTP internally

Kong Route — Full Anatomy

A Route is the incoming request matcher. Every field in a Route controls how Kong decides whether an incoming HTTP request belongs to this route, and how it transforms the request before forwarding it.

FieldValue in our setupWhere it comes from / What it does
name api1-test-route You define this. Used as the identifier in Admin API calls. Convention: {app}-{path}-route.
paths ["/test"] The URL prefix Kong listens on at the Proxy (kong-proxy.local). Incoming requests starting with this prefix are matched to this route.
strip_path false (test) / true (bgtest) false = forward the full path to the upstream (e.g. /test → /test). true = strip the matched prefix before forwarding (e.g. /bgtest → /, then Service.path appends /test).
protocols ["http","https"] Kong accepts requests on both HTTP and HTTPS for this route. Default when not specified.
hosts null If set, Kong only matches requests with this Host header (e.g. vault-route has vault.local). Null means match any host.
methods null If set, restricts the route to specific HTTP methods (GET, POST, etc). Null means all methods are accepted.
service.id 20128166-0d92-... The Kong Service this route belongs to. Set automatically when you create the route via /services/{name}/routes. This is how Kong knows where to forward the matched request.
path_handling v0 Kong's path normalization mode. v0 is the default and handles path prefix stripping correctly for most cases.
preserve_host false false = Kong replaces the Host header with the upstream's hostname. true = Kong preserves the original client Host header when forwarding.
regex_priority 0 When multiple routes could match the same path, the one with the highest priority wins. Default 0 is fine when paths don't overlap.
request_buffering true Kong buffers the full request body before forwarding. Set to false for streaming/large uploads.
response_buffering true Kong buffers the full upstream response before sending to client. Set to false for streaming responses (SSE, etc).
https_redirect_status_code 426 HTTP status code returned when Kong redirects HTTP → HTTPS. 426 = Upgrade Required.
ℹ strip_path explained with an example

Route path: /bgtest, strip_path: true, Service path: /test
Client sends: GET http://kong-proxy.local/bgtest
Kong strips /bgtest → forwards to upstream as: GET /test

Route path: /test, strip_path: false, no Service path
Client sends: GET http://kong-proxy.local/test
Kong preserves path → forwards as: GET /test

Kong Upstream & Targets

An Upstream is Kong's built-in load balancer. It acts as a virtual hostname that sits between a Kong Service and multiple backend K8s services. Targets are the individual backend entries with weights.

ObjectFieldValueMeaning
Upstream name api1-upstream This is used as the host value in the Kong Service. Must match exactly.
algorithm round-robin How Kong distributes requests. Options: round-robin, least-connections, consistent-hashing.
Target target api1-stable-service.default.svc.cluster.local:80 Full K8s DNS name of the backend service including port. Must be reachable from within the cluster.
weight 70 / 30 Relative weight. Kong distributes requests proportionally. 70+30=100 means 70% stable, 30% canary.
✓ Key architectural difference

When a Kong Service has host: api1-upstream — it routes through the Upstream load balancer (Canary strategy).
When a Kong Service has host: api1-blue-service.default.svc.cluster.local — it routes directly to one K8s service (Blue/Green strategy).

Canary Deployment
Gradually shift traffic from a stable version to a new version by adjusting Kong Upstream target weights. No redeployment required to change traffic ratios. Instant rollback by setting weights back.

  • Instant rollbackPATCH weight back to 0

  • K8s resources2 Deployments + 2 Services
  • Switch commandPATCH target weight
  • Route changesNone — route unchanged

Step 1 — Build Docker Images

PowerShell — from your project root
docker build -t api1-app:stable .
docker build -t api1-app:canary .
⚠ Same codebase, different tags

Both images are built from the same code. The difference is the environment variables injected via YAML (APP_VERSION, DEPLOYMENT_SLOT). The image tag only needs to differ if you want to ensure the images are built from genuinely different code.

Step 4 — Configure Kong (one-time)

1
Create the Upstream
PowerShell
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        name      = "api1-upstream"
        algorithm = "round-robin"
    } | ConvertTo-Json)
2
Add Stable Target (weight 70)
PowerShell
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        target = "api1-stable-service.default.svc.cluster.local:80"
        weight = 70
    } | ConvertTo-Json)
3
Add Canary Target (weight 30)
PowerShell
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        target = "api1-canary-service.default.svc.cluster.local:80"
        weight = 30
    } | ConvertTo-Json)
4
Point Kong Service to the Upstream (PATCH existing service)
PowerShell
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-service" `
    -Method PATCH -ContentType 'application/json' `
    -Body (@{ host = "api1-upstream" } | ConvertTo-Json)
ℹ Why PATCH and not POST?

api1-service already existed and api1-test-route already pointed to it. Instead of deleting and recreating — which would also delete the route — we simply update the host field. The route is untouched.

06 — Operations
Traffic Management
Shift canary traffic weight in phases without any redeployment. The Upstream weights are live — changes take effect immediately.

Get Target IDs (run once, save the output)

PowerShell
$targets  = Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets"
$stableId = ($targets.data | Where-Object { $_.target -like "*stable*" }).id
$canaryId = ($targets.data | Where-Object { $_.target -like "*canary*" }).id
Write-Host "Stable: $stableId"
Write-Host "Canary: $canaryId"

Canary Promotion Phases

PhaseStable WeightCanary WeightUse When
Initial9010First exposure — minimal blast radius
Phase 27030No errors observed at 10%
Phase 35050Confidence growing — equal split
Full Promotion0100Canary is now the stable version
Rollback1000Errors detected — instant revert
PowerShell — Phase helper function
function Set-CanaryWeights($stableWeight, $canaryWeight) {
    Invoke-RestMethod `
        -Uri "http://kong-admin.local/upstreams/api1-upstream/targets/$stableId" `
        -Method PATCH -ContentType 'application/json' `
        -Body (@{ weight = $stableWeight } | ConvertTo-Json)

    Invoke-RestMethod `
        -Uri "http://kong-admin.local/upstreams/api1-upstream/targets/$canaryId" `
        -Method PATCH -ContentType 'application/json' `
        -Body (@{ weight = $canaryWeight } | ConvertTo-Json)

    Write-Host "Weights updated — Stable: $stableWeight | Canary: $canaryWeight"
}

# Usage:
Set-CanaryWeights 70 30    # Phase 2
Set-CanaryWeights 50 50    # Phase 3
Set-CanaryWeights 0 100    # Full promotion
Set-CanaryWeights 100 0    # Emergency rollback

Kong Admin — Modify

PowerShell
# Point service to upstream (canary mode)
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-service" `
    -Method PATCH -ContentType 'application/json' `
    -Body (@{ host = "api1-upstream" } | ConvertTo-Json)

# Switch to Green (blue/green cutover)
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-bg-service" `
    -Method PATCH -ContentType 'application/json' `
    -Body (@{ host = "api1-green-service.default.svc.cluster.local" } | ConvertTo-Json)

# Rollback to Blue
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-bg-service" `
    -Method PATCH -ContentType 'application/json' `
    -Body (@{ host = "api1-blue-service.default.svc.cluster.local" } | ConvertTo-Json)

# Delete route
Invoke-RestMethod -Uri "http://kong-admin.local/routes/api1-bg-route" -Method DELETE

# Delete service
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-bg-service" -Method DELETE
