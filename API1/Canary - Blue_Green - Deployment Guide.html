Naming Conventions
Every name in this system flows from a single source. Understanding this chain prevents the most common mistakes — especially the image tag mismatch that causes pods to fail on startup.

⚠ The #1 Mistake

Building with docker build -t api1-app:stable . but writing image: api1-app:v1 in the YAML. Kubernetes looks for the exact tag. If it doesn't match a local image and imagePullPolicy: Always is set, it tries to pull from Docker Hub — and fails. Always use imagePullPolicy: IfNotPresent for local images.

03 — Foundation
Initial Kong Setup
Before any deployment strategy can work, you must create the core Kong Services and Routes that will be used throughout this guide. This section creates the foundational objects that all later steps assume exist.

What we will create

Kong ServicePurposeInitial HostRoutePath
api1-serviceCanary (upstream)api1-stable-service.default.svc.cluster.localapi1-test-route/test
api1-bg-serviceBlue/Green cutoverapi1-blue-service.default.svc.cluster.localapi1-bg-route/bgtest
ℹ Prerequisites

Before running these commands, ensure you have deployed at least the stable and blue Kubernetes resources. If you haven't, refer to the Canary Build & Deploy YAMLs section for the stable Deployment/Service, and the Blue/Green Build & Deploy YAMLs section for the blue Deployment/Service. For now, we assume those are already running.

1
Create Kong Service for Canary
▶ PowerShell POST /services — create api1-service pointing to stable backend 
Invoke-RestMethod -Uri "http://kong-admin.local/services" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        name     = "api1-service"
        host     = "api1-stable-service.default.svc.cluster.local"
        port     = 80
        protocol = "http"
    } | ConvertTo-Json)
2
Create Kong Route for Canary (/test)
▶ PowerShell POST /services/api1-service/routes — create api1-test-route 
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-service/routes" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        name       = "api1-test-route"
        paths      = @("/test")
        strip_path = $false
    } | ConvertTo-Json)
3
Create Kong Service for Blue/Green
▶ PowerShell POST /services — create api1-bg-service pointing to blue backend 
Invoke-RestMethod -Uri "http://kong-admin.local/services" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        name     = "api1-bg-service"
        host     = "api1-blue-service.default.svc.cluster.local"
        port     = 80
        protocol = "http"
        path     = "/test"           # app only exposes /test; route will strip /bgtest
    } | ConvertTo-Json)
4
Create Kong Route for Blue/Green (/bgtest)
▶ PowerShell POST /services/api1-bg-service/routes — create api1-bg-route with strip_path 
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-bg-service/routes" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        name       = "api1-bg-route"
        paths      = @("/bgtest")
        strip_path = $true
    } | ConvertTo-Json)
5
Verify both routes work
▶ PowerShell Test Canary route and Blue/Green route 
# Should return version = "v1-stable", slot = "stable"
Invoke-RestMethod "http://kong-proxy.local/test" | ConvertTo-Json

# Should return version = "v1-blue", slot = "blue"
Invoke-RestMethod "http://kong-proxy.local/bgtest" | ConvertTo-Json
✓ Foundation ready

You now have the two essential Kong Services and their corresponding Routes. All subsequent Canary and Blue/Green operations will build on top of this base. The Canary strategy will later patch the api1-service to point to an Upstream, and the Blue/Green strategy will patch the api1-bg-service to point to green.

04 — Reference
Kong Concepts Deep Dive
Kong's three core objects — Service, Route, and Upstream — work together to handle every request. This section documents exactly where each value comes from in your local setup.

Kong Service

A Kong Service is the bridge between a Kong Route (incoming request matcher) and the actual upstream backend. In this setup, a Service either points to a direct K8s service DNS name (Blue/Green) or to a Kong Upstream name (Canary).

Service → Direct K8s (Blue/Green)
name
api1-bg-serviceYou define this. Referenced when creating routes.
host
api1-blue-service.default.svc.cluster.localK8s Service name + namespace + cluster suffix. This is the PATCH target during cutover.
port
80From K8s Service spec.ports[].port
protocol
httpKestrel runs on plain HTTP internally
path
/testSet when Kong route path ≠ app path (e.g. /bgtest → /test)
Service → Upstream (Canary)
name
api1-serviceYou define this. Referenced when creating routes.
host
api1-upstreamThe Kong Upstream name — NOT a DNS name. Kong resolves it via its upstream registry.
port
80Must match the port in each Upstream Target.
protocol
httpKestrel runs on plain HTTP internally

Kong Route — Full Field Anatomy

FieldValue in this setupMeaning & Source
nameapi1-test-routeYou define this. Used as identifier in Admin API calls. Convention: {app}-{path}-route.
paths["/test"]URL prefix Kong listens on at the Proxy. Requests starting with this prefix are matched to this route.
strip_pathfalse (test) / true (bgtest)false = forward full path as-is. true = strip matched prefix before forwarding, then append Service.path.
protocols["http","https"]Kong accepts both HTTP and HTTPS for this route. Default when not specified.
hostsnullIf set, Kong only matches requests with this Host header. Null = match any host.
headers{"X-Canary": ["true"]}If set, route only matches requests containing this header/value pair. Used for header-based routing.
methodsnullIf set, restricts to specific HTTP methods. Null = all methods accepted.
service.idauto-setThe Kong Service this route belongs to. Set automatically when you create the route via /services/{name}/routes.
regex_priority0 (default) / 10 (header route)Higher priority wins when multiple routes match. Header routes must have higher priority than the base route.

Kong Upstream & Targets

ObjectFieldValueMeaning
Upstreamnameapi1-upstreamUsed as the host in the Kong Service. Must match exactly.
algorithmround-robinHow Kong distributes requests. Options: round-robin, least-connections, consistent-hashing.
Targettargetapi1-stable-service.default.svc.cluster.local:80Full K8s DNS name including port. Must be reachable from within the cluster.
weight70 / 30Relative weight. Kong distributes proportionally. 70+30=100 → 70% stable, 30% canary.
05 — Strategy
Canary Deployment
Gradually shift traffic from a stable version to a new version by adjusting Kong Upstream target weights. No redeployment required to change traffic ratios. Instant rollback by resetting weights.
What it solves
  • Blast radius controlsmall % of users hit new version
  • Real traffic validationobserve errors & latency in prod
  • Incremental confidenceshift 10→30→50→100 gradually
  • Instant rollbackPATCH weight back to 0
Kong mechanism
  • Kong objectUpstream + Targets
  • K8s resources2 Deployments + 2 Services
  • Switch commandPATCH target weight
  • Route changesNone — existing route untouched

Step 4 — Configure Kong (one-time)

1
Create the Upstream
▶ PowerShell POST /upstreams — create api1-upstream 
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        name      = "api1-upstream"
        algorithm = "round-robin"
    } | ConvertTo-Json)
2
Add Stable Target (weight 70)
▶ PowerShell POST /upstreams/api1-upstream/targets — stable at 70% 
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        target = "api1-stable-service.default.svc.cluster.local:80"
        weight = 70
    } | ConvertTo-Json)
3
Add Canary Target (weight 30)
▶ PowerShell POST /upstreams/api1-upstream/targets — canary at 30% 
Invoke-RestMethod -Uri "http://kong-admin.local/upstreams/api1-upstream/targets" `
    -Method POST -ContentType 'application/json' `
    -Body (@{
        target = "api1-canary-service.default.svc.cluster.local:80"
        weight = 30
    } | ConvertTo-Json)
4
Point Kong Service to the Upstream (PATCH existing service)
▶ PowerShell PATCH /services/api1-service — redirect to upstream 
Invoke-RestMethod -Uri "http://kong-admin.local/services/api1-service" `
    -Method PATCH -ContentType 'application/json' `
    -Body (@{ host = "api1-upstream" } | ConvertTo-Json)
ℹ Why PATCH and not POST?

api1-service was created in the Initial Kong Setup section. PATCHing only the host field leaves the existing route (api1-test-route) intact — no downtime, no route recreation.

06 — Strategy
Blue/Green Deployment
Run two full environments simultaneously. Blue is live. Green is deployed and privately validated. A single PATCH command switches all traffic instantly — zero downtime, immediate rollback capability.
What it solves
  • Zero-downtime releasesno dropped connections
  • Full env validationtest green before going live
  • Instant rollbackone PATCH command
  • Predictable switchoverall-or-nothing, no partial state
Kong mechanism
  • Kong objectService host URL swap
  • K8s resources2 Deployments + 2 Services
  • Switch commandPATCH service host
  • No Upstream neededdirect K8s DNS target

07 — Strategy
Header-Based Traffic Routing
The safest and most widely adopted production routing pattern. A higher-priority route with a header condition intercepts targeted requests — without modifying any existing routes or services. Your stable path is never touched.
✓ Core Principle

Add a new route with a header condition and higher regex_priority. Do not modify existing routes. Kong evaluates routes by priority — the header-conditioned route wins for matching requests, and all other traffic falls through to the original stable route.

Why this is production-safe
  • No route changesexisting routes untouched
  • No service changeszero config drift
  • Zero downtimeadditive, not disruptive
  • Instant rollbackDELETE the header route
  • Targeted testingonly flagged clients hit new version
Industry adoption
  • Kongheaders + regex_priority
  • IstioVirtualService header match
  • AWS ALBheader-based routing rules
  • TraefikHeader middleware routing
  • NGINX$http_x_* variables

▶ PowerShell Set-CanaryWeights helper function 
function Set-CanaryWeights($stableWeight, $canaryWeight) {
    Invoke-RestMethod `
        -Uri "http://kong-admin.local/upstreams/api1-upstream/targets/$stableId" `
        -Method PATCH -ContentType 'application/json' `
        -Body (@{ weight = $stableWeight } | ConvertTo-Json)

    Invoke-RestMethod `
        -Uri "http://kong-admin.local/upstreams/api1-upstream/targets/$canaryId" `
        -Method PATCH -ContentType 'application/json' `
        -Body (@{ weight = $canaryWeight } | ConvertTo-Json)

    Write-Host "✓ Weights updated — Stable: $stableWeight | Canary: $canaryWeight"
}
